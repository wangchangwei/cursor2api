### IDEA-1: 上游健康探测 + 预防性断开

**Title**: 请求前上游健康探测（Pre-stream Probe）

**Description**:
在发起流式请求前，先向上游 Cursor API 发送一个轻量探测请求（如单轮 "ping" 消息），验证连接可用且能返回有效内容。若探测返回空/拒绝，则在请求前就切换备用节点或返回降级响应，避免下游等到超时。

**Key Assumption**:
上游在探测时有内容返回能力。Probe 请求本身可能被拒绝或限流。

**Potential Impact**:
- 高：将断流问题提前到请求前解决，下游零感知
- 减少无效请求对上游的压力

**Implementation Hint**:
在 `handler.ts` / `openai-handler.ts` 中，`sendCursorRequest` 之前增加一个 2-3 秒超时的探测调用，根据探测结果决定是否继续真实请求。探测结果写入 `cursorClient.ts` 的连接健康缓存。

### IDEA-2: 流式内容速度监控 + 提前中断重试

**Title**: 字节速度窗口监控（Byte Velocity Monitor）

**Description**:
在流式传输过程中维护一个滚动窗口（如 5 秒），统计该窗口内的有效数据字节数。若窗口内有效字节数低于阈值（如 < 50 bytes），判定为"流速异常"，立即触发探针重试，而不是等到整流结束才发现响应过短。

**Key Assumption**:
正常 Cursor API 流式响应在任意 5 秒窗口内必有数据产生。

**Potential Impact**:
- 高：将检测时机从"末尾判断"提前到"过程判断"，节省下游等待时间
- 与现有 keepalive 机制正交，可叠加

**Implementation Hint**:
在 `handler.ts` 的 `processHybridDelta` 回调中增加速度监控逻辑。超过阈值时调用 `buildShortResponseRetryRequest` 触发重试，超时则直接进入最终降级。

### IDEA-3: 结构化合流响应（SSE Multiplexing）

**Title**: SSE 多路复用结构化消息帧（SSE Structured Frame）

**Description**:
将 SSE 流从纯文本/工具块扩展为结构化消息帧，包含 `type`、`sequence`、`length_hint` 字段。下游可据此判断是否已收到足够内容。空响应时，中间件主动发送一个 `type: "meta"` 帧告知当前流状态（如 "upstream_refused"），下游据此决定是否等待。

**Key Assumption**:
下游客户端能解析并响应新增的 meta 帧类型。

**Potential Impact**:
- 中：需要下游客户端配合更新，但向后兼容（未知帧类型可忽略）
- 大幅提升问题定位能力

**Implementation Hint**:
在 `handler.ts` 和 `openai-handler.ts` 的流式写入逻辑中，嵌入 meta 帧结构。定义新的 SSE event type 如 `cursor-stream-meta`。

### IDEA-4: 短路熔断器（Upstream Circuit Breaker）

**Title**: 短路熔断器模式（Circuit Breaker for Cursor API）

**Description**:
在应用层维护一个熔断状态机（Closed → Open → Half-Open）。当某个时间窗口内（如 60 秒）短响应/空响应次数超过阈值，判定 Cursor API 当前状态异常，切换到熔断模式：直接返回降级响应而不调用上游，防止下游长时间等待。

**Key Assumption**:
短响应是 Cursor API 端问题而非偶发，重试无意义。

**Potential Impact**:
- 高：彻底消除因上游不稳定导致的下游断流
- 需要精心设计阈值，避免误判

**Implementation Hint**:
在 `cursor-client.ts` 中实现熔断器状态机。结合现有的 `MAX_REFUSAL_RETRIES` 逻辑，增加全局窗口计数。

### IDEA-5: 连接上下文保持重试（Context-Preserving Retry）

**Title**: 连接上下文保持重试（Connection Context Preservation）

**Description**:
现有重试机制是"请求级"重试——重试时新建请求，上下文通过追加 user 消息传递。问题在于：下游 TCP 连接已建立但无数据，持续等待会触发网关超时。改进方案：在重试期间保持向下游发送 keepalive，同时利用已积累的部分响应作为上下文（如有），让探针重试携带更多语义信息。

**Key Assumption**:
部分响应（即使很短）可能包含有效语义，只是被截断了。

**Potential Impact**:
- 高：重试成功率提升，下游等待体验改善
- 减少"探针重试后仍无可见正文"的边界情况

**Implementation Hint**:
在 `buildShortResponseRetryRequest` 中，将 `fullResponse`（哪怕只有几字符）作为 system prompt 的补充上下文传入，而非仅用固定的探针文本。

### IDEA-6: 有效负载最小保证（Minimum Payload Guarantee）

**Title**: 最小有效负载保证（Minimum Viable Payload）

**Description**:
当前 keepalive 机制发送 `: keepalive\n\n`，但这只是保持连接，不提供内容信号。改进方案：在流开始时强制发送至少一个有效的"引言" chunk（如 "Processing your request..."），确保下游收到第一个可见内容后再开始可能的长等待。若后续流速过慢，keepalive 阶段性地发送有意义的进度信息而非纯注释。

**Key Assumption**:
下游在收到第一个有效内容后会启动解析，即使后续为空也会等待。

**Potential Impact**:
- 中：改善用户体验，减少"白屏"感知
- 不解决根本问题，但缓解症状

**Implementation Hint**:
在 `handler.ts` 的 `handleDirectTextStream` 和工具模式 `processHybridDelta` 开头，增加一个 Promise.race 超时（3s），超时前无内容则写入引导文本。

### IDEA-7: 降级响应语义化（Semantic Fallback Response）

**Title**: 语义化降级响应（Semantic Degraded Response）

**Description**:
当前降级文本是 `"Let me proceed with the task."`，问题是下游可能将其视为有效内容而非错误标记。改进方案：降级响应携带明确的语义标记（如 SSE `event: error`），让下游能识别这是降级响应并做相应处理（如显示错误提示、请求用户确认）。

**Key Assumption**:
下游客户端能处理 error 类型事件。

**Potential Impact**:
- 高：下游从"蒙在鼓里"变为"知情等待"
- 配合 IDEA-3 的结构化帧效果最佳

**Implementation Hint**:
在 `handler.ts` 和 `openai-handler.ts` 中，将硬编码的 `"Let me proceed with the task."` 替换为 SSE error 事件格式，并在最终 `res.end()` 前发送错误标记。

### IDEA-8: 响应完整性契约（Response Completeness Contract）

**Title**: 响应完整性契约（Completeness Receipt）

**Description**:
在每个流结束时（`res.end()` 前），发送一个结构化的"收据"事件，包含：`total_bytes`、`stop_reason`、`has_tool_calls`、`is_degraded`。下游收到收据后验证：如果 `total_bytes` 极低且 `is_degraded=true`，说明响应异常。这将问题检测从"响应中"延迟到"响应后"，但提供了明确的契约。

**Key Assumption**:
下游客户端会在收据阶段做最终验证。

**Potential Impact**:
- 中：提供可审计的完整性记录，便于事后分析
- 对实时修复帮助有限，但提升可观测性

**Implementation Hint**:
在 `handler.ts` 流结束逻辑中添加 `cursor-stream-receipt` SSE 事件类型，携带完整性元数据。
